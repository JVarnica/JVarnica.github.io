## Building a Research Agent


This is part of the execuchat project: a log of the issues that came up while building the research agent used in the app. Before building the graph need to think what data will be sent to the Android UI. You want to be able to track progress, so each time there is a new event it is sent. For example, 'generating queries', 'search_queries', 'summarize_docs', 'extract_claims', 'reflect', 'plan report', 'write section' and 'stitch report'.

The template was generated by Claude, with debugging done by me so it works end to end on the server. Thispost is about the main issues faced and how they were solved to create a working chatbot. First will show the architecture and events log can be found here [events1_log.txt]. 

All code is on the research-agent repo on github [here], the how is this built and main blog is [here].

The architecture of the graph is:
-generate queries -> search_one_query(parrallel/Send) -> summarize_one_doc(parallel/Send) \
-> extract_claims -> Reflect -> plan_report -> write_section(parallel/Send) -> stitch report

### Extracting Claims

Another issue occured when extracting claims, the system was passing the whole accumulated doc_summaries pool to the claim extractor, instead of just the new relevant summaries. Consequently, loop 1 will derive claims from both the new summaries and  the old summaries from loop 0. This creates a batch with near identical claims, so by the time planning comes there are now dozens of near duplicate claims given to the model, this bloats context length and hinders the answer. So the trick is to track which documents have already contributed to extracted claims. This is simple as the Claim class has source_doc_ids field which is just the doc_ids in Doc_summary, so the new relevant doc_summaries are just the doc_ids not in the covered_doc_ids set.  Now how does doc_id map to source_doc_ids, well the model does it, in summary prompt there is a clear rule for this, and as the model sees doc_id:xxxx very clearly in summaries_json knows what to do.
```
#Before
relevant = [s for s in state["doc_summaries"] if s.relevant]
summaries_json = json.dumps([s.model_dump() for s in relevant])

#Now
covered_doc_ids = {did for c in state.get("claims", []) for did in c.source_doc_ids}
new_relevant = [s for s in state["doc_summaries"] if s.relevant and s.doc_id not in covered_doc_ids]
summaries_json = json.dumps([s.model_dump() for s in new_relevant])

...part of extract prompt
- Cite source documents by their doc_id in source_doc_ids.
- Multiple sources supporting the same claim → high confidence; single source → medium; conflicting → low.
- Do NOT include generic background or filler. Only specific claims that help answer the question.

Summaries:
{summaries_json}"""
```

This has been successful as the claims extracted on events_log1.txt was 17 claims on iteration 0, and 30 claims iteration 1; whereas in logs2 it is 9 and 6 claims. The difference is also the research question one is how LLM works and the second log is how Byzantium collasped more interesting and will be the research question from now on.
